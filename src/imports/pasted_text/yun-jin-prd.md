1 概述
1.1 术语
授权素材台账	对非遗/云锦元素建立来源、授权范围、商用条件、改造限制、登记信息、工艺限制等结构化台账。
云锦设计副驾	给设计师和客户经理使用的 AI 辅助设计工作台，用于需求录入、方向生成、调参与效果图输出。
提案即成交系统	在 20-30 分钟内完成需求录入、方向生成、产品效果图展示、预算说明、授权说明和留资/下单的提案系统。
确权与复用台账	围绕一次创作过程形成版本链、证据链、授权链、复用链的交易级素材履历系统。
工艺约束	指云锦工艺可落地所需遵循的色数、纹样复杂度、材质、织造限制、适用品类等业务规则。
主题包 / 系列包	可按主题、品类、品牌联名等方式组织的复用资产包，用于后续复用、授权和经营。
B2B 私域工作台	面向设计师、品牌客户、博物馆、景区、礼赠、软装、策展等机构用户使用的封闭式工作台。
规则引擎	将授权规则、工艺规则、提示词模板、审核约束等配置化并在流程中调用的系统能力。
版本证据包	记录输入、素材选择、修改、审核、授权结果的归档集合，用于存证、争议处理和复用经营。
交易级素材履历	将每次创作的输入、修改、审核、授权与复用结果结构化沉淀，作为后续经营资产。

1.3 背景和价值
本项目不是横向素材平台，也不以从零训练基础大模型为目标，而是面向云锦与非遗设计场景，构建“可授权素材台账 + 云锦设计副驾 + 提案即成交系统 + 确权与复用台账”的行业工作流产品。
现状中，设计师虽然能在全网找到大量图片，但难以判断“是否能放心用、是否能继续改、是否能落到云锦工艺上”；客户希望在沟通现场先看到高保真方向图和落地效果图，不愿等待数天；项目交付后，设计成果又往往无法沉淀为可复用、可再授权、可分账的资产。
• 为什么要做：把散落在素材、设计、销售、授权、存证、复用之间的链路整合为一条可管理、可经营、可验证的业务闭环。
• 不做的影响：设计提案继续依赖人工低效沟通；授权风险无法前置控制；作品难沉淀为数字资产；每单项目都从零开始，无法形成复利。
• 做成后的价值：提案时长显著缩短；客户留单率提升；授权边界清晰；设计成果可形成主题包、系列包、联名包等复用资产；组织沉淀交易级素材履历。
1.4 需求概述
本期需求面向 B2B 私域工作台，覆盖从“线索进入—需求录入—素材检索—方案生成—现场提案—留资下单—版本归档—授权发放—复用入池”的核心业务链。产品以现有 Foundation Model 为底座，叠加自有授权数据集、行业知识库、规则引擎和工作流界面。
• 本次需求要解决的问题：让销售、设计师和运营/法务能够在一套系统内完成素材可用性判断、方案快速生成、提案成交流转和确权复用沉淀。
• 本次需求边界（做）：素材授权台账、需求录入、方向生成、风格调参、产品/场景效果图、提案页、预算与授权说明、版本归档、授权流水、复用资产池、管理后台。
• 本次需求边界（不做）：开放式 C 端市场、开放 marketplace、从零训练通用大模型、复杂 ERP/财务系统、全自动法律文书生成、对外 API 商业开放。

2.2 功能1：工作台首页与任务中心
2.2.1 场景描述
工作台首页是所有角色进入系统后的第一触点，用于统一承载待办任务、最近项目、快捷入口、经营看板和消息提醒。其目标是让用户在 1 分钟内识别当前最重要任务并进入下一步操作。
2.2.2 用户角色
• 设计师
• 销售/客户经理
• 运营/法务
• 管理员
2.2.3 前置条件
• 用户已完成登录并具有组织与项目权限。
• 组织已配置菜单权限、项目权限与红点规则。
2.2.4 事件流程
1. 基本事件流程：
N0010  系统根据用户角色加载待办任务、最近项目、快捷入口和经营看板。
N0020  用户点击待办任务进入对应流程，如待审核提案、待补充授权信息、待跟进客户。
N0030  用户通过快捷入口发起新建提案、新建项目、导入素材或查看复用资产。
N0040  用户处理完成任务后，系统更新红点状态并返回首页。
N0050  结束。
2. 异常事件流程：
E0010  若用户无对应菜单权限，则隐藏菜单，并在分享链接进入时提示“无访问权限”。
E0020  若首页数据加载超时超过 3 秒，显示骨架屏；超过 30 秒显示可取消并允许重试。
E0030  若待办任务已失效，点击后提示原因并提供返回路径；结束。

2.3 功能2：可授权素材台账
2.3.1 场景描述
素材台账不是普通图库，而是“可用性系统”。设计师不再先去全网找图，而是先在系统中查找授权清晰、可检索、可衍生、可落到工艺上的云锦/非遗元素。
2.3.2 用户角色
• 设计师
• 销售/客户经理（只读检索）
• 运营/法务（审核与维护）
• 管理员
2.3.3 前置条件
• 已存在经过授权梳理的素材记录。
• 系统已配置基础标签体系与授权规则。
2.3.4 事件流程
1. 基本事件流程：
N0010  用户输入关键词或选择标签（题材、颜色、工艺、授权状态、品类等）发起检索。
N0020  系统返回元素列表，并对不可商用、不可改造、限特定客户等情况进行醒目标识。
N0030  用户进入元素详情查看来源、传承人/合作方、授权范围、登记信息、历史案例和工艺限制。
N0040  用户点击“加入当前方案”或“加入主题包”，系统进行一次授权校验。
N0050  结束。
2. 异常事件流程：
E0010  若元素授权状态不完整，则禁止加入当前方案，并提示补全字段。
E0020  若元素与当前客户/地域/品类不匹配，则系统提示限制条件，不允许继续。
E0030  若检索无结果，提供相似标签、推荐元素与人工申请补录入口；结束。

2.4 功能3：云锦设计副驾
2.4.1 场景描述
设计副驾服务于设计师和客户经理，不替代设计师，而是帮助其在沟通现场快速形成情绪提案板、多个方向方案、高保真效果图与设计说明。
2.4.2 用户角色
• 设计师
• 销售/客户经理
2.4.3 前置条件
• 当前项目已创建。
• 至少已选择一个客户场景或品类。
• 系统可访问素材台账与工作流引擎。
2.4.4 事件流程
1. 基本事件流程：
N0010  用户在左侧面板录入客户类型、场景、品类、预算、审美偏好和风格参数。
N0020  系统调用素材匹配、风格控制、图案重组与高保真生成能力，输出 3-8 个方向。
N0030  用户选择一个方向后，通过传统度/现代度、礼品感/时尚感、复杂度/可生产性等维度继续调参。
N0040  系统重新生成情绪板、产品图、场景图和设计说明。
N0050  用户将选中的方向提交到提案中心；结束。
2. 异常事件流程：
E0010  若输入信息不足以生成方案，系统在面板中高亮缺失字段并禁用“生成”。
E0020  若生成结果不满足工艺约束，则标注“需人工修订”并给出原因。
E0030  若生成失败或超时，系统保留已输入参数，允许重试或切换模型模板；结束。

2.5 功能4：提案即成交系统
2.5.1 场景描述
提案中心承接设计副驾输出，用于在客户面前 20-30 分钟内完成需求确认、方案展示、预算区间说明、授权说明以及留资/下单，是本期最重要的成交效率模块。
2.5.2 用户角色
• 销售/客户经理
• 设计师（协同展示）
• 客户（查看展示）
2.5.3 前置条件
• 项目中至少存在一个可展示方向。
• 已完成初步授权校验。
2.5.4 事件流程
1. 基本事件流程：
N0010  销售在提案中心选择客户与项目，系统展示当前可用方案。
N0020  销售依次向客户展示方向图、产品效果图、场景图和设计说明。
N0030  系统同步展示预算区间、授权说明、交付周期和风险提示。
N0040  客户选择感兴趣方向后，销售在系统中完成留资、预约二次沟通或创建订单。
N0050  系统写入成交结果，并将项目状态流转到“跟进中”或“已签单”；结束。
2. 异常事件流程：
E0010  若方案授权限制不满足当前客户，系统在展示页显著提示“仅供方向参考，不可直接授权使用”。
E0020  若预算字段缺失，系统禁止提交下单，但允许留资与预约。
E0030  若网络中断，系统保留当前展示状态和客户输入，恢复后可继续；结束。

2.6 功能5：确权与复用台账
2.6.1 场景描述
确权与复用台账不是单纯的“出图后去登记”，而是把每次创作变成未来可复用、可授权、可分账的资产。系统需记录输入、修改、审核、授权和后续二次授权的全过程。
2.6.2 用户角色
• 运营/法务
• 设计师（查看本人项目）
• 销售（查看客户授权结果）
• 管理员
2.6.3 前置条件
• 项目已形成至少一个版本结果。
• 系统已接入证据归档/存证能力（内部或外部）。
2.6.4 事件流程
1. 基本事件流程：
N0010  系统在每次生成、编辑、审核、提案、成交时写入版本记录。
N0020  运营/法务进入确权台账查看版本时间轴、素材选择记录、审核记录和客户信息。
N0030  系统基于项目状态生成证据包，并支持存证、登记编号回写和授权流水管理。
N0040  运营将已确认可复用的方案打标加入主题包/系列包/复用资产池。
N0050  后续发生二次授权时，系统记录授权对象、期限、地域、品类和分账规则；结束。
2. 异常事件流程：
E0010  若证据包关键字段缺失，系统禁止发起存证/登记并提示补齐。
E0020  若发生授权冲突（如同一元素超出限制使用），系统高亮告警并要求人工处理。
E0030  若外部存证接口失败，系统保留本地证据包并进入待重试队列；结束。

5 非功能需求
5.1 懒人原则
• 60 秒原则：高频任务（检索、生成、保存、刷新、导出）应提供明显反馈；首页首屏、素材检索首屏结果、提案页切换应以 3 秒以内为体验目标。
• 4 步原则：从首页发起新提案、从素材台账加入方案、从提案中心完成留资，均应控制在 4 步以内。
• 做任务原则：工作台首页必须承接红点/角标任务；任务完成后状态实时回写。
• 爬楼梯原则：表单出错时定位到首个错误字段；必须具备取消、返回、草稿保存和重试机制。
• 保底线原则：外网接口一律使用 HTTPS；上线时不得硬编码敏感信息；权限不足时菜单隐藏；系统必须通过隐私和合规审核。
• 上线 10 要件：新增、删除、修改、触发类操作必须记录日志；关键页面具备仪表盘、权限、水印、双语适配和异常工单入口。
5.2 安全性需求（必选）
• 基于组织、角色、项目、客户四级授权控制页面与数据访问范围。
• 素材、方案、证据包、授权流水均需记录访问日志和操作日志。
• 下载、导出、分享链接需支持时效控制、水印和权限校验。
• 关键数据（授权范围、登记编号、客户信息、分账规则）在存储和传输过程中加密处理。
• 管理后台规则变更必须保留审计日志，并支持版本回滚。
5.3 性能需求
场景	指标	说明
首页加载	首屏核心组件 3 秒内展示	允许骨架屏；超过 30 秒需支持取消与重试。
素材检索	首次结果 3 秒内返回	复杂筛选可异步补全，但需先返回首批结果。
方向生成	标准任务 60-120 秒内完成	根据模型能力允许排队，但需展示进度与预计完成状态。
提案页切换	1 秒内完成方向切换	切换仅重载差异内容，不整体刷新。
证据包生成	3 分钟内完成	外部存证失败需进入待重试队列。
5.4 可靠性与可维护性
• 关键流程需支持草稿保存、断点恢复、失败重试。
• 外部接口（存证/登记）失败时不影响内部证据包生成和归档。
• 新增、删除、修改、触发类行为记录日志，便于线上追查问题。
• 规则和模板均应配置化，避免写死在代码中。
• 异常发生时记录环境参数、请求参数、用户、项目、时间和链路ID。
5.5 扩展性与兼容性
• 未来可扩展第二品类（如秦淮灯彩、绒花、城市纹样），当前架构需支持多品类标签、规则、模板扩展。
• 未来可扩展 API 与交易模块，当前数据模型需预留授权对象、交易对象和分账规则字段。
• 浏览器兼容性：优先支持 Chrome 最新两个大版本；兼容 Edge。
• 界面预留中英双语能力，但当前内容以中文为主。
5.6 可测试性
• 每个 P0 功能至少维护最小化自动化测试用例集。
• 关键 Given/When/Then 验收用例需沉淀为接口测试和 UI 冒烟测试。
• 流程节点需具有可观测日志，便于测试复盘链路。
6 开发实施建议
• 建议按“素材台账 → 设计副驾 → 提案中心 → 确权台账 → 管理后台”顺序迭代，以保证最短业务闭环先跑通。
• MVP 阶段可先以封闭试点客户和内部团队为目标，不追求大规模并发与开放接入。
• 研发侧应先统一项目、方案、素材、授权、版本五大核心实体的数据模型，再展开页面与接口开发。
• 测试侧优先围绕 P0 链路建立端到端回归用例：新建项目、检索素材、生成方向、现场提案、留资、归档。
• 设计侧优先完善五个关键页面的高保真交互和状态说明，确保异常态、空态、加载态、权限态齐备。
